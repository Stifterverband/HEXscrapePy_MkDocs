LSF-Funktionen

Diese Seite beschreibt die Funktionen aus hexscrapepy.lsf_scraper.

Rollen der Funktionen

Öffentlich empfohlen

Funktion Rolle Kurzbeschreibung
scrape_base_data_lsf Basisdaten erhebt Veranstaltungslisten eines oder mehrerer Semester
scrape_lsf_veranstaltungen Detaildaten ergänzt Detailinformationen zu vorhandenen URL_Event-Links
scrape_lsf_all_semesters Wrapper führt den vollständigen LSF-Workflow pro Semester aus

Modular nutzbar

Funktion Rolle Kurzbeschreibung
select_semester Navigation wechselt im LSF zum gewünschten Semester
extract_veranstaltungstitel_lsf Extraktion liest den sichtbaren Titel einer Veranstaltung
extract_grunddaten_lsf Extraktion liest strukturierte Grunddaten aus der Detailseite
extract_termine_lsf Extraktion liest Terminblöcke einer Veranstaltung
extract_dozenten_lsf Extraktion extrahiert Dozentinnen und Dozenten
extract_zugeordnete_personen_lsf Extraktion extrahiert weitere zugeordnete Personen
extract_studiengaenge_lsf Extraktion extrahiert zugeordnete Studiengänge
extract_module_lsf Extraktion extrahiert zugeordnete Module
extract_einrichtung_lsf Extraktion extrahiert Einrichtungen oder organisatorische Zuordnungen
extract_weitere_angaben_lsf Extraktion liest weitere Angaben als strukturierte Felder
extract_lerninhalt_lsf Extraktion zieht Lerninhalte aus den weiteren Angaben
extract_kommentar_lsf Extraktion zieht Kommentare aus den weiteren Angaben
fetch_lsf_course_html Abruf lädt die rohe HTML-Seite einer Veranstaltung
fetch_lsf_course_html_expanded Abruf lädt die Seite mit aufgeklappten Toggle-Bereichen

Intern oder technisch

Funktion Rolle Kurzbeschreibung
clean_text Helper normalisiert Freitext für Export und Parsing
normalize_url Helper reduziert LSF-URLs auf stabile Vergleichswerte
_scrape_via_path Workflow interner Basislauf über die Baumstruktur
_scrape_via_search Workflow interner Basislauf über die Suchmaske
_find_event_tables Parser identifiziert Tabellen mit Veranstaltungslisten
_parse_event_table Parser liest einzelne Veranstaltungszeilen in Ergebnis-Dictionaries
_find_lsf_section_table Parser findet Bereichstabellen robust über mehrere Signale
_needs_expanded_lsf_fetch Steuerung entscheidet über Playwright-Fallback
_load_lsf_results_json Resume lädt bestehende Zwischenergebnisse
_write_lsf_results_json Resume schreibt Zwischenergebnisse zurück
_build_lsf_detail_frame Merge reduziert Detaildaten auf mergefähige Spalten

Typischer Ablauf

Ein vollständiger LSF-Lauf sieht meist so aus:

  1. scrape_lsf_all_semesters() als Wrapper
  2. intern scrape_base_data_lsf() für die Basisdaten
  3. anschließend scrape_lsf_veranstaltungen() für die Detailseiten
  4. einzelne extract_*-Parser lesen strukturierte Felder aus dem HTML

extract_weitere_angaben_lsf(soup)

Extrahiert alle Felder aus Weitere Angaben zur Veranstaltung.

Die LSF-Ansicht kapselt diesen Bereich teils in auf-/zuklappbare Container (z.B. veranst_slide_inhalt). Für das Parsing ist das unkritisch, solange die Tabelle im HTML enthalten ist. Diese Funktion liest deshalb die Tabellenzeilen generisch aus und liefert eine einzige Zeile mit allen vorhandenen Feldern zurück.

extract_zugeordnete_personen_lsf(soup)

Extrahiert die zugeordneten Personen als verschachtelte Tabelle.

Diese Funktion ist für LSF-Blöcke wie Zugeordnete Person oder Verantwortliche Dozenten gedacht. Das Ergebnis wird bewusst als verschachtelte Tabelle zurückgegeben, damit die Struktur aus dem HTML möglichst erhalten bleibt.

Rückgabespalten der inneren Tabelle: - zugeordnete_person - zustaendigkeit

fetch_lsf_course_html_expanded(url, timeout_ms=30000)

Lädt eine LSF-Veranstaltungsseite mit aufgeklappten Toggle-Bereichen.

Einige LSF-Installationen rendern Bereiche wie Inhalt, Termine oder Zugeordnete Person erst nach einem Klick auf veranst_toggle_*. Dieser Fallback nutzt Playwright, öffnet alle vorhandenen Toggle-Links und gibt danach das resultierende HTML zurück.

scrape_lsf_all_semesters(start_url, semester_liste, output_dir, url_column='URL_Event', method='path', include_html=True, html_column='html')

Führt den vollständigen LSF-Workflow pro Semester aus.

Pro Semester werden nacheinander: 1. Basisdaten mit scrape_base_data_lsf geladen, 2. Duplikate in der Basisdatei entfernt, 3. Detaildaten mit scrape_lsf_veranstaltungen gescrapt, 4. semester-spezifische CSV/JSON-Dateien gespeichert.

Dateien im output_dir: - Pro Semester ein Unterordner <semester>/ - Darin base_data_<semester-token>.csv und course_data_<semester-token>.json

Parameters:

Name Type Description Default
start_url str

LSF-Start-URL der Universität.

 required
semester_liste

Liste der gewünschten Semesterbezeichnungen.

 required
output_dir str

Zielordner für alle erzeugten Dateien.

 required
url_column str

Spalte mit Veranstaltungs-URLs für das Detail-Scraping.

 'URL_Event'
method str

Erhebungsmethode für den Basislauf ("path" oder "search").

 'path'
include_html bool

Wenn True, wird das rohe HTML jeder Kursdetailseite im Ergebnis gespeichert.

 True
html_column str

Name der Spalte für das rohe HTML.

 'html'

Returns:

Type Description
dict

Dict mit Semestername als Key und gescraptem DataFrame als Value.

Öffentliche Paket-API

Diese API wird über hexscrapepy direkt exportiert:

hexscrapepy

HEXscrapePy package.

Die öffentliche API wird per Lazy-Import bereitgestellt. Dadurch bleibt ein einfaches import hexscrapepy leichtgewichtig und zieht optionale schwerere Abhängigkeiten wie Playwright erst dann nach, wenn eine entsprechende Funktion wirklich genutzt wird.
